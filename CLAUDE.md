# Dotfiles

Max's macOS dotfiles, managed with GNU Stow.

## How Stow Works

Each top-level directory is a **stow package** that mirrors the target file structure relative to `~`. Stow creates symlinks from `~` into this repo. Since everything is symlinked, editing files in this repo directly changes the active config.

The `--dotfiles` flag in `.stowrc` means files named `dot-*` become `.*` when symlinked. For example:

- `zsh/dot-zshrc` → `~/.zshrc`
- `git/dot-gitconfig` → `~/.gitconfig`
- `p10k/dot-p10k.zsh` → `~/.p10k.zsh`
- `brewfile/dot-config/brewfile/Brewfile` → `~/.config/brewfile/Brewfile`
- `worktrunk/scripts/worktrunk/wt-zellij-tab` → `~/scripts/worktrunk/wt-zellij-tab`

To add a new config:

1. Create a new directory at the repo root (e.g. `aerospace/`)
2. Inside it, mirror the path structure relative to `~` using `dot-` prefix for dotfiles
3. Add `stow <dirname>` to `setup.sh`
4. Run `stow <dirname>` to create the symlink

Re-stow is only needed after adding new files/packages, not after edits.

- **revdiff** — Terminal UI for reviewing diffs/files with inline annotations; emits annotations to stdout on quit for piping into AI agents. Supports git, hg, and jj.
  - Config: `revdiff/dot-config/revdiff/config` (INI). Customized from the `--dump-config` template: `compact` (GitHub-style hunks + context), `line-numbers`, `word-diff`, `untracked` shown, monochrome `✎` annotation marker, and `theme = everforest`. The theme is a stowed file — `revdiff/dot-config/revdiff/themes/everforest` (`chroma-style = evergarden` + Everforest Dark Hard `color-*`, matching Ghostty/Zellij). The 8 bundled themes are written to `~/.config/revdiff/themes/` at runtime (everforest is symlinked in alongside them) — see Things to Know. Keybindings are a stowed delta file — `revdiff/dot-config/revdiff/keybindings` (template via `revdiff --dump-keys`) — remapping `[`/`]` → prev/next annotation and `(`/`)` → prev/next hunk (deltas on the defaults).
  - No man page — use `revdiff --help`
  - Online docs / repo: https://github.com/umputun/revdiff
  - Claude Code integration (two plugins from the `revdiff` marketplace = GitHub `umputun/revdiff`): `revdiff` provides the `/revdiff` overlay-review command (`/revdiff`, `/revdiff main`, `/revdiff HEAD~1`, `/revdiff --staged`, …), and `revdiff-planning` auto-opens a review when Claude exits plan mode. Install: `claude plugin marketplace add umputun/revdiff`, then `claude plugin install revdiff@revdiff` and `claude plugin install revdiff-planning@revdiff`.
  - Zellij keybind `Alt r` (`zellij/dot-config/zellij/config.kdl`) opens a floating `revdiff` reviewing uncommitted changes **instantly** — no `/revdiff` model round-trip to open the panel. Bare `revdiff` only (no smart ref-detection; that's the skill's `detect-ref.sh`, not the binary). On quit, revdiff auto-saves annotations to `~/.config/revdiff/history/<repo>/`; run `/rd` to apply them (see next bullet).
  - `/rd` skill (`~/.claude/skills/rd/SKILL.md`) applies the latest history annotations after you close the overlay — user-only (`disable-model-invocation: true`, so Claude never auto-runs it). **Not stowed** — machine-local like the plugins. Keybind/usage cheat: `navi/dot-local/share/navi/cheats/revdiff.cheat` (open navi with `Ctrl /` in Zellij).

  - Script: `storybook/scripts/storybook/storybook-pxds` → `~/scripts/storybook/storybook-pxds`. On PATH via `dot-zshenv`'s `~/scripts/*(N-/)` glob, so the keybind runs it as a bare `Run "storybook-pxds"`. No brew package — Storybook comes from the rosetta repo's `node_modules`.
  - Keybind: `Alt s` in `zellij/dot-config/zellij/config.kdl`.
  - Behaviour: **summon-only** (`Alt s` shows; never hides — dismiss with the native floating toggle `Alt f`). Floating panes are per-tab, so each worktree tab gets its own Storybook. "Running here?" is detected from `zellij action list-panes --json`, scoped to the current tab via `$ZELLIJ_PANE_ID`. Hardcoded to rosetta pxds for now (the `--cwd`/package can become dynamic later).

  - Repo: https://github.com/KiryuuLight/zellij-attention

## Adding a New Tool

Checklist when adding a new CLI tool or app to this setup:

1. **Install** — `brew install <name>` (or `brew install --cask <name>` for GUI apps)
2. **Brewfile** — Add the entry to `brewfile/.config/brewfile/Brewfile` in alphabetical order under the correct tap/section
3. **Stow package** — If the tool has config files, create a stow package:
   - Create `<name>/` at the repo root, mirroring the path structure relative to `~`
   - Use `dot-` prefix for dotfiles (e.g. `dot-config` becomes `.config`)
   - Add `stow <name>` to `setup.sh` (alphabetical within the stow block)
   - Run `stow <name>` to create the symlinks
4. **Local docs** — Find local documentation and add it to the Tools section below:
   - Check for man pages: `man -w <name>` (gives the path if one exists)
   - Check for bundled docs: look in `/opt/homebrew/share/doc/<name>/`, the app bundle, or run `<name> --help`
   - Check for completions: `ls /opt/homebrew/share/zsh/site-functions/_<name>`
5. **Tools entry** — Add an entry to the **Tools** section in this file with: a short description, config path (if stow-managed), and local doc paths found above
6. **Things to Know** — If the tool has ordering constraints, gotchas, or interactions with other tools, add a bullet to the **Things to Know** section

Skip steps that don't apply (e.g. no stow package needed for a tool with no config).

## Things to Know

- Zellij autostart **must** stay at the very top of `.zshrc` (before p10k instant prompt) — it launches zellij and exits the outer shell, so nothing after it runs in that shell.
- The `.zshrc` sources `~/.localrc` at the end for machine-specific config that shouldn't be committed.
- `zoxide` init **must** stay last in `.zshrc` — it overrides `cd` and the `chpwd` hook.
- P10k instant prompt **must** stay at the top of `.zshrc`.
- Worktrunk shell integration **must** come before zoxide in `.zshrc` — it needs the shell wrapper for `wt switch` to change directories, but doesn't conflict with zoxide's `cd` override.
- Non-interactive Node access depends on `fnm default` — `.zshenv` prepends `~/.local/share/fnm/aliases/default/bin`, so changing the global default should be done with `fnm default <version>`.
- Plugin order in `dot-zsh_plugins.txt` matters: `fzf-tab` before `autosuggestions`.
- **Stow uses tree folding** — directories like `~/.config/zellij` may be symlinks to the stow source, not real directories. Running `rm` on files inside them deletes the stow source files. Never `rm` files in stow-managed paths to "clean up before restowing." If you need to restow, use `stow -R <package>` instead.
- **Linear → worktree integration** has two non-obvious constraints. (1) It assumes a **single live Zellij session** — `linear-work` picks the most recent non-EXITED session, since Linear launches it with no `ZELLIJ` env of its own. (2) Linear *generates and may rewrite* `~/.linear/coding-tools.json`; it's stowed for reproducibility, but a Linear settings change can write through (or replace) the symlink. The prompt reaches Claude only because `linear-work` exports `WT_CLAUDE_PROMPT`, which the `post-switch` hook inherits — Zellij panes take the **server's** env, not the caller's, so the prompt is injected into the layout as a `claude` argument rather than passed via the environment.
- **zellij-attention icon slots are semantically inverted from their names.** The plugin only exposes `completed_icon` and `waiting_icon`. We map `completed_icon` (✓) to Claude being *idle / your turn* (`Notification:idle_prompt`) and `waiting_icon` (●) to Claude *awaiting a permission decision* (`Notification:permission_prompt`) — so "completed" here does **not** mean "task finished". The plugin **load** lives in this repo's `config.kdl`, but the **triggering hooks** live in the global `~/.claude/settings.json` (outside this repo); changing one without the other breaks the badges.
- **Dev-server "open in editor" → Helix.** `.zshrc` sets `LAUNCH_EDITOR`/`REACT_EDITOR` to `~/scripts/helix/hx-open` (`helix/scripts/helix/hx-open`), which opens the file in a Helix pane in a new Zellij tab. This is **not** redundant with `EDITOR=hx`: Storybook/Vite/webpack route through the `launch-editor` npm package, which scans running processes and picks a known GUI editor (e.g. VS Code) *before* it ever consults `$VISUAL`/`$EDITOR` — only `LAUNCH_EDITOR`/`REACT_EDITOR` jump that queue. A dedicated tab (not direct spawn) is required because launch-editor would otherwise run the TUI with the dev server's inherited stdio and hijack its pane. launch-editor passes only the file path (no line number) for editors it doesn't special-case, so the cursor lands at the top of the file. Requires the dev server to be running inside Zellij (it inherits `$ZELLIJ`); outside Zellij the script no-ops.
- **revdiff's Claude Code integration needs no sandbox carve-out here, and its plugin state is not dotfiles-managed.** The `/revdiff` launcher detects the terminal by priority (agterm, tmux, **Zellij**, …, ghostty, iTerm2, vterm) and the first match wins. Because `.zshrc` always starts Zellij, the **Zellij** CLI launcher is selected — which the Claude Code Bash sandbox doesn't block — so the `permissions.excludedCommands: ["*/launch-revdiff.sh*"]` workaround (only needed for the Ghostty/iTerm2 launchers, which use sandbox-blocked `osascript`) is **not** required. The plugins + marketplace live in `~/.claude/plugins/*.json` (plus `enabledPlugins` in `~/.claude/settings.json`), which is **not** stowed or in `setup.sh`; reinstall on a fresh machine with the `claude plugin …` commands in the revdiff Tools entry above.
- **revdiff writes into its own config dir at runtime, so it's stowed with `--no-folding`.** On first run revdiff writes the 8 bundled themes into `~/.config/revdiff/themes/`. If stow folded `~/.config/revdiff` into a symlink (which it does on a fresh machine, where the dir doesn't exist at stow time), those generated theme files would land inside this repo. `stow --no-folding revdiff` (in `setup.sh`) keeps `~/.config/revdiff` a real directory and symlinks only the `config` file — a concrete case of the tree-folding hazard above.
- **The `Alt s` Storybook keybind is summon-only by necessity, and several of its choices are load-bearing — don't "simplify" them.** The hard constraint: a one-key launch-*or*-toggle would need to read the *prior* floating-pane visibility, but the dispatcher must run in a pane, and every pane placement is bad for that — `floating` forces the layer visible (so prior visibility is unreadable), and `in_place` *can* read it but suspends/replaces your focused work pane (this was the original bug). The escape hatch is **summon-only**: by only ever ending in "shown", we never need to read prior visibility, so the dispatcher can run `floating` and leave your tiled work pane untouched. Hence: hiding is the **native `Alt f`** toggle, not `Alt s`. Other non-obvious bits: (1) it runs **`vp exec storybook`**, not `pnpm` — Zellij command panes get only the `~/.zshenv` PATH, where `vp` lives (rosetta is a Vite+ workspace) and `pnpm` does not; (2) "is Storybook in *this* tab?" is scoped via **`$ZELLIJ_PANE_ID`** (the dispatcher's own pane id → its `tab_id`), **not** `is_focused` — `PaneInfo.is_focused` is per-layer-per-tab, so picking the first focused pane lands on the wrong worktree tab; (3) detection matches the launched pane's `--name "storybook"` exactly, excluding the dispatcher's own id.
