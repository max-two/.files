# Dotfiles

Max's macOS dotfiles, managed with GNU Stow.

## How Stow Works

Each top-level directory is a **stow package** that mirrors the target file structure relative to `~`. Stow creates symlinks from `~` into this repo. Since everything is symlinked, editing files in this repo directly changes the active config.

The `--dotfiles` flag in `.stowrc` means files named `dot-*` become `.*` when symlinked. For example:

- `zsh/dot-zshrc` ‚Üí `~/.zshrc`
- `git/dot-gitconfig` ‚Üí `~/.gitconfig`
- `p10k/dot-p10k.zsh` ‚Üí `~/.p10k.zsh`
- `brewfile/dot-config/brewfile/Brewfile` ‚Üí `~/.config/brewfile/Brewfile`
- `worktrunk/scripts/worktrunk/wt-zellij-tab` ‚Üí `~/scripts/worktrunk/wt-zellij-tab`

To add a new config:

1. Create a new directory at the repo root (e.g. `aerospace/`)
2. Inside it, mirror the path structure relative to `~` using `dot-` prefix for dotfiles
3. Add `stow <dirname>` to `setup.sh`
4. Run `stow <dirname>` to create the symlink

Re-stow is only needed after adding new files/packages, not after edits.

## Adding a New Tool

Checklist when adding a new CLI tool or app to this setup:

1. **Install** ‚Äî `brew install <name>` (or `brew install --cask <name>` for GUI apps)
2. **Brewfile** ‚Äî Add the entry to `brewfile/.config/brewfile/Brewfile` in alphabetical order under the correct tap/section
3. **Stow package** ‚Äî If the tool has config files, create a stow package:
   - Create `<name>/` at the repo root, mirroring the path structure relative to `~`
   - Use `dot-` prefix for dotfiles (e.g. `dot-config` becomes `.config`)
   - Add `stow <name>` to `setup.sh` (alphabetical within the stow block)
   - Run `stow <name>` to create the symlinks
4. **Local docs** ‚Äî Find local documentation and add it to the Tools section below:
   - Check for man pages: `man -w <name>` (gives the path if one exists)
   - Check for bundled docs: look in `/opt/homebrew/share/doc/<name>/`, the app bundle, or run `<name> --help`
   - Check for completions: `ls /opt/homebrew/share/zsh/site-functions/_<name>`
5. **Tools entry** ‚Äî Add an entry to the **Tools** section in this file with: a short description, config path (if stow-managed), and local doc paths found above
6. **Things to Know** ‚Äî If the tool has ordering constraints, gotchas, or interactions with other tools, add a bullet to the **Things to Know** section

Skip steps that don't apply (e.g. no stow package needed for a tool with no config).

## Things to Know

- Zellij autostart **must** stay at the very top of `.zshrc` (before p10k instant prompt) ‚Äî it launches zellij and exits the outer shell, so nothing after it runs in that shell.
- The `.zshrc` sources `~/.localrc` at the end for machine-specific config that shouldn't be committed.
- `zoxide` init **must** stay last in `.zshrc` ‚Äî it overrides `cd` and the `chpwd` hook.
- P10k instant prompt **must** stay at the top of `.zshrc`.
- Worktrunk shell integration **must** come before zoxide in `.zshrc` ‚Äî it needs the shell wrapper for `wt switch` to change directories, but doesn't conflict with zoxide's `cd` override.
- Non-interactive Node access depends on `fnm default` ‚Äî `.zshenv` prepends `~/.local/share/fnm/aliases/default/bin`, so changing the global default should be done with `fnm default <version>`.
- Plugin order in `dot-zsh_plugins.txt` matters: `fzf-tab` before `autosuggestions`.
- **Stow uses tree folding** ‚Äî directories like `~/.config/zellij` may be symlinks to the stow source, not real directories. Running `rm` on files inside them deletes the stow source files. Never `rm` files in stow-managed paths to "clean up before restowing." If you need to restow, use `stow -R <package>` instead.
- **Linear ‚Üí worktree integration** has two non-obvious constraints. (1) It assumes a **single live Zellij session** ‚Äî `linear-work` picks the most recent non-EXITED session, since Linear launches it with no `ZELLIJ` env of its own. (2) Linear *generates and may rewrite* `~/.linear/coding-tools.json`; it's stowed for reproducibility, but a Linear settings change can write through (or replace) the symlink. The prompt reaches Claude only because `linear-work` exports `WT_CLAUDE_PROMPT`, which the `post-switch` hook inherits ‚Äî Zellij panes take the **server's** env, not the caller's, so the prompt is injected into the layout as a `claude` argument rather than passed via the environment.
- **zellij-attention icon slots are semantically inverted from their names.** The plugin only exposes `completed_icon` and `waiting_icon`. We map `completed_icon` (‚úì) to Claude being *idle / your turn* (`Notification:idle_prompt`) and `waiting_icon` (‚óè) to Claude *awaiting a permission decision* (`Notification:permission_prompt`) ‚Äî so "completed" here does **not** mean "task finished". The plugin **load** lives in this repo's `config.kdl`, but the **triggering hooks** live in the global `~/.claude/settings.json` (outside this repo); changing one without the other breaks the badges.
- **Dev-server "open in editor" ‚Üí Helix.** `.zshrc` sets `LAUNCH_EDITOR`/`REACT_EDITOR` to `~/scripts/helix/hx-open` (`helix/scripts/helix/hx-open`), which opens the file in a Helix pane in a new Zellij tab. This is **not** redundant with `EDITOR=hx`: Storybook/Vite/webpack route through the `launch-editor` npm package, which scans running processes and picks a known GUI editor (e.g. VS Code) *before* it ever consults `$VISUAL`/`$EDITOR` ‚Äî only `LAUNCH_EDITOR`/`REACT_EDITOR` jump that queue. A dedicated tab (not direct spawn) is required because launch-editor would otherwise run the TUI with the dev server's inherited stdio and hijack its pane. launch-editor passes only the file path (no line number) for editors it doesn't special-case, so the cursor lands at the top of the file. Requires the dev server to be running inside Zellij (it inherits `$ZELLIJ`); outside Zellij the script no-ops.
- **revdiff's Claude Code integration needs no sandbox carve-out here, and its plugin state is not dotfiles-managed.** The `/revdiff` launcher detects the terminal by priority (agterm, tmux, **Zellij**, ‚Ä¶, ghostty, iTerm2, vterm) and the first match wins. Because `.zshrc` always starts Zellij, the **Zellij** CLI launcher is selected ‚Äî which the Claude Code Bash sandbox doesn't block ‚Äî so the `permissions.excludedCommands: ["*/launch-revdiff.sh*"]` workaround (only needed for the Ghostty/iTerm2 launchers, which use sandbox-blocked `osascript`) is **not** required. The plugins + marketplace live in `~/.claude/plugins/*.json` (plus `enabledPlugins` in `~/.claude/settings.json`), which is **not** stowed or in `setup.sh`; reinstall on a fresh machine with the `claude plugin ‚Ä¶` commands in the revdiff Tools entry above.
- **revdiff writes into its own config dir at runtime, so it's stowed with `--no-folding`.** On first run revdiff writes the 8 bundled themes into `~/.config/revdiff/themes/`. If stow folded `~/.config/revdiff` into a symlink (which it does on a fresh machine, where the dir doesn't exist at stow time), those generated theme files would land inside this repo. `stow --no-folding revdiff` (in `setup.sh`) keeps `~/.config/revdiff` a real directory and symlinks only the `config` file ‚Äî a concrete case of the tree-folding hazard above.
- **The `Alt n` notes and `Alt r` revdiff binds share the `zj-summon` dispatcher, and the `Alt s` Storybook bind (`storybook-pxds`) is a separate copy of the same design (stacked, not floating); its choices are load-bearing ‚Äî don't "simplify" them.** The hard constraint: a one-key launch-*or*-toggle would need to read the *prior* floating-pane visibility, but the dispatcher must run in a pane, and every pane placement is bad for that ‚Äî `floating` forces the layer visible (so prior visibility is unreadable), and `in_place` *can* read it but suspends/replaces your focused work pane (this was the original bug). The escape hatch is **summon-only**: by only ever ending in "shown", we never need to read prior visibility, so the dispatcher can run `floating` and leave your tiled work pane untouched. Hence: hiding is the **native `Alt f`** toggle, not `Alt s`. Other non-obvious bits: (1) it runs **`vp exec storybook`**, not `pnpm` ‚Äî Zellij command panes get only the `~/.zshenv` PATH, where `vp` lives (rosetta is a Vite+ workspace) and `pnpm` does not; (2) "is Storybook in *this* tab?" is scoped via **`$ZELLIJ_PANE_ID`** (the dispatcher's own pane id ‚Üí its `tab_id`), **not** `is_focused` ‚Äî `PaneInfo.is_focused` is per-layer-per-tab, so picking the first focused pane lands on the wrong worktree tab; (3) detection matches the launched pane's `--name "storybook"` exactly, excluding the dispatcher's own id; (4) it's **worktree-aware** via the dispatcher's inherited cwd ‚Äî a `Run { floating true }` pane opens in the focused pane's cwd (same as the `Alt r` revdiff bind), so the script does `git rev-parse --show-toplevel` from there and runs that worktree's `packages/pxds`, no-opping unless it's `@healthbyro/pxds`. If Zellij ever stops inheriting the focused pane's cwd, the resolution silently no-ops (safe, but the keybind would appear dead).
- **The `Alt n` notes scratchpad uses the shared `zj-summon` dispatcher (also used by the `Alt r` revdiff bind) ‚Äî the summon-only / per-tab / worktree-aware design and the "don't simplify these" caveats above apply identically.** Two notes-specific facts: (1) detection keys on the pane's `terminal_command` (= the keybind `Run` target, `notes`), which is fixed at pane birth and immune to OSC retitling ‚Äî this replaced an earlier `title`-based match that broke for apps which overwrite their title via OSC (revdiff ‚Üí `rd: <ref>`, claude ‚Üí its task summary). Helix here happens not to retitle, but the mechanism no longer depends on that. (2) The note's gitignore coverage comes from the **stowed global excludes** at `~/.config/git/ignore` ‚Äî git reads that path automatically as the XDG default with **no `core.excludesfile` set**. It pre-existed as a real file on this machine and was replaced by the stow symlink (`stow git`); a fresh machine gets it straight from `stow git`. `*.local`/`*.local.*` is promoted there from rosetta's own `.gitignore`, so the pattern (and thus `notes.local.md` staying out of `git status`) now holds in every repo, not just rosetta. Auto-cleanup works because a *gitignored* untracked file does not block `git worktree remove` (verified).
- **Worktree tab naming splits durable state from live-derived state ‚Äî keep that split; and the async design is load-bearing (don't make it synchronous or reach for `--bare`).** Guiding principle: *persist only what we own and that's stable; resolve anything zellij owns live.* The **only** persisted state is the AI label, via `worktree-name` (TSV `path‚á•label` at `~/.cache/worktree-tab-names`; `set` appends so concurrent namers can't corrupt it, reads are last-wins, `clear` on worktree removal) ‚Äî a label never goes stale, so a lost entry just falls back to `project-list`'s derived name, never a correctness bug. Tab identity is **never** persisted: `worktree-tab-id <path>` resolves a worktree's tab live by pane CWD (the single resolver shared by `zellij-tab-open`, `zellij-tab-close`, `worktree-tab-namer`), because zellij gives no events to invalidate a cached tab id ‚Äî a stale id could hit the wrong tab, and re-validating one against live state costs the same as resolving fresh. Flow: `zellij-tab-open` creates the tab immediately with the deterministic fallback label (repo + Linear-prefix-stripped, 25-char-truncated branch), then spawns `worktree-tab-namer` **detached**; the namer asks `claude -p --model haiku` for a label from the **raw** branch (Haiku strips the `username-team` prefix and keeps the ticket number itself, so `project-list`'s strip regex isn't duplicated), `worktree-name set`s it, then renames the live tab via `rename-tab --tab-id` (tab resolved by `worktree-tab-id`, so **no focus change**). Runs **once per worktree** ‚Äî later opens read the cache and are instant. **Why async:** a headless `claude -p` costs ~10s here and it's irreducible ‚Äî OAuth is mandatory (managed settings reject any API key: *"a non-OAuth credential cannot satisfy the org pin"*), so the fast `--bare`/`CLAUDE_CODE_SIMPLE=1` path is unavailable (stripping MCP/settings/hooks/slash-commands didn't help either); a synchronous call would freeze tab creation for ~10s. **Why CWD dedup, not name:** the async rename means a tab's name may still be the interim fallback, so a name match could miss it and open a duplicate. Regenerate a label with `worktree-name clear <path>` (next open re-runs the namer).
- **Worktree tab *teardown* is two-phase across `wt remove`'s hooks, and both the split and the phase assignment are load-bearing ‚Äî don't collapse it to a single post-remove CWD lookup or move the close to pre-remove.** `[pre-remove]` runs `zellij-tab-close capture '{{ worktree_path }}'` and `[post-remove]` runs `zellij-tab-close '{{ worktree_path }}'`. The tab is identified by pane CWD, but **`wt remove` renames the worktree into `<repo>/.bare/wt/trash/<name>-<ts>` before `post-remove` fires**, which changes the panes' CWDs ‚Äî so a CWD lookup at post-remove matches nothing. (This is exactly what silently broke teardown when close switched from name-based to CWD-based matching: a tab's *name* survives the rename, its panes' *CWD* does not.) So we **capture** the tab id(s) at pre-remove ‚Äî worktree still in place, CWD still matches ‚Äî into `~/.cache/wt-tab-close/<cksum-of-path>`, and **close by that id** at post-remove (then delete the capture). The close must stay in post-remove: `Alt x` runs `project-close` as a floating pane in the *current* tab, so closing that tab during pre-remove (before the removal runs) would kill the in-flight `wt remove` and leave the worktree half-removed; by post-remove the removal is already done, so closing ‚Äî even project-close's own tab ‚Äî is safe. Capturing at pre-remove is safe because `wt` runs its dirty-check first, so `pre-remove` only fires when the removal will actually proceed (a dirty worktree that fails removal never fires it). All verified end-to-end.
- **A newly `stow`-created `~/scripts/<tool>` dir isn't on a *running* Zellij session's PATH until that session restarts.** The zellij server captures `~/.zshenv`'s `~/scripts/*(N-/)` glob once at startup, and command panes inherit that frozen env (they don't re-source `.zshenv`) ‚Äî so a brand-new script-backed keybind (e.g. `Alt r` ‚Üí `revdiff-summon` right after `stow revdiff`) silently no-ops in the current session and works only in a new one. Symptom to recognize: the keybind "does nothing" ‚Äî its pane opens, hits command-not-found, and `close_on_exit` removes it before the error is visible. (Verified: the server PATH lacked `~/scripts/revdiff` immediately after stow, while a fresh `zsh` resolved it.)
